# Cloud Observability & Dynamic Workload Simulator

This repository demonstrates an Infrastructure as Code (IaC) approach using **Ansible** and **Terraform** to automatically provision an observability and load-testing environment on **AWS EC2 (Ubuntu)** instances. 

The project deploys **Docker**, telemetry monitoring agents (**Prometheus Node Exporter**), and a custom **chaotic workload simulator in Python** managed as a native Linux daemon (`systemd`). It is designed to emulate real-world enterprise traffic patterns and stress-test monitoring stacks and automated alerting systems.

---

## Project Architecture

The infrastructure deployment is structured using **Ansible Roles** to enforce modularity, code reusability, and strict **idempotency**.

| Ansible Role | Core Responsibility | Technical Implementation Details |
|---|---|---|
| **`docker`** | Container engine provisioning | Installs `docker-ce` from official repositories, manages GPG keys, and adds the `ubuntu` user to the `docker` group to enable rootless CLI operations. |
| **`node_exporter`** | System metrics telemetry | Deploys Node Exporter as a background service (port `9100`). Uses **Jinja2 templates** (`.j2`) and **Handlers** to ensure the service restarts *only* when configuration changes occur. |
| **`target_script`** | Production workload simulation | Deploys a custom Python resource stressor managed by **Systemd**. Implements automated Handler triggers to gracefully reload the daemon whenever updates to the Python script are applied. |

---

## Chaotic Workload Simulator (`stress_script.py`)

Unlike traditional stress-testing tools that flatline CPU and RAM at 100% statically, this project implements a dynamic traffic generator that models the chaotic, organic behavior of a live production server.

### Engineered Load Mechanics
1. **Dynamic Duty Cycle:** CPU consumption is regulated across 100ms time windows, alternating between active computation and sleep phases independently on every physical CPU core.
2. **Random Walk:** Baseline system resource consumption drifts organically over time to simulate natural fluctuations in user demand.
3. **High-Frequency Jitter:** Millisecond-to-millisecond variations are injected into CPU and RAM allocation, eliminating artificial, flat lines in telemetry dashboards.
4. **Traffic Micro-bursts:** Emulates unexpected traffic spikes (such as DDoS attacks or sudden flash sales) by driving CPU utilization to 85%–95% for randomized intervals of 12 to 30 seconds.

## Prerequisites:

Before running this project, ensure you have the following tools and configurations set up on your local control node:

* **Cloud Provider Access:** An active AWS account with appropriate permissions to provision EC2 instances, VPCs, and Security Groups.
* **AWS CLI & Credentials:** Configured locally (via `aws configure`, environment variables, or `~/.aws/credentials`) so both Terraform and Ansible can interact with your AWS environment.
* **Terraform:** [Installed](https://developer.hashicorp.com/terraform/install) (v1.0+ recommended) for Infrastructure as Code provisioning.
* **Ansible:** [Installed](https://docs.ansible.com/ansible/latest/installation_guide/index.html) (v2.10+ recommended) for configuration management.
* **SSH Key Pair:** An AWS EC2 Key Pair (`.pem` file) created in your target AWS region and available locally with read-only permissions (`chmod 400 /path/to/key.pem`) to allow Ansible SSH access.

## Step-by-Step Deployment Guide

Follow this sequence to provision the infrastructure from scratch, configure the servers, and start visualizing the telemetry data.

1. Clone this repository to your local machine and navigate into the project directory.

2. Deploy the EC2 instances:
- cd Terraform/
- terraform init
- terraform plan
- terraform apply --auto-approve

Important: Once terraform apply completes, copy or note down the Public IPs of the newly launched EC2 instances (Target EC2 and Monitoring EC2) displayed in the Terraform terminal outputs.

3. Navigate to the Ansible/ directory. Open the inventory.ini file and paste the Public IPs generated by Terraform into their respective groups:

```text
[target]
PUBLIC-IP ansible_user=ubuntu

[monitoring]
PUBLIC-IP ansible_user=ubuntu

```

4. On Ansible/ directory, run:

- ansible-playbook -i inventory.ini site.yaml

5. Open your web browser and navigate to the Prometheus web UI hosted on your Monitoring EC2 instance:

- http://<PUBLIC_IP_MONITORING_INSTANCE>:9090

6. Click on the Graph tab at the top of the Prometheus dashboard.

7. Paste the following PromQL queries into the search bar and click Execute to see the live impact of the chaotic stress script:

- To view live CPU Utilization:
- 100 - (avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[1m])) * 100)


- To view live RAM Allocation:
- (node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes) / 1024 / 1024 / 1024

## CI/CD
The project also contains a jenkins file for a CI/CD pipeline. These are the stages and their description:

* **1. Code Checkout:**Pulls the latest source code from the GitHub repository into the Jenkins workspace to initiate the pipeline.

* **2. DevSecOps (Static Security Analysis):**Runs SAST scans using Bandit (Python) and Checkov (Ansible IaC) to catch security flaws before deployment.

* **3. Quality Gate (CI: Lint & Syntax):**Validates Python script compilation and checks Ansible YAML syntax (--syntax-check) to prevent runtime failures.

* **4. Dry Run (Risk-Free Simulation):**Simulates playbook execution (--check --diff) on AWS in read-only mode to preview changes without modifying servers.

* **5. AWS Deployment (CD):**Applies real configurations to EC2 instances over SSH, provisioning packages, Docker, and system services.

* **6. Smoke Tests (SRE Health Checks):**Audits active systemd service daemons and verifies telemetry endpoints (Node Exporter & Prometheus) return HTTP 200 OK.
